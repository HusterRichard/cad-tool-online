# CAD 场景交付

## 概述

使用这个 skill 交付一个 CadToolOnline 场景，从证据收集到代码验证完整收口。若用户只输入一个 `SCxx`，默认按完整场景工作流执行，除非用户显式缩小范围。

## 快速开始

1. 运行 `python .codex/skills/cad-scene-delivery/scripts/lookup_scene.py SC12`，将 `SC12` 替换为当前场景 ID。
2. 阅读 `.codex/skills/cad-scene-delivery/references/repo-conventions.md`。
3. 读取全局 PRD 中该场景对应行、映射到的帮助页 HTML、该场景最新的现有文档，以及最相关的实现文件。
4. 默认执行完整链路：
   - 创建或更新场景 PRD
   - 创建或更新场景 PLAN
   - 创建或更新场景 QA 计划
   - 实现功能代码
   - 增加或更新回归测试
   - 运行定向验证
5. 所有项目 Markdown 交付文档统一使用简体中文，并以 UTF-8 保存。

如果用户明确要求只做其中一层，例如“只写 PRD”或“只做代码”，则只执行对应子集。

## 工作流

### 1. 解析场景

- 先运行 `scripts/lookup_scene.py`。
- 确认以下信息：
  - 场景名称
  - 所属模块
  - 帮助页路径
  - 画板 ID
  - 现有最新的 PRD、PLAN、QA 文档
- 如果该场景已有同类文档，优先更新最新文件。只有在不存在匹配文件，或用户明确要求新修订版时，才创建新的日期文件。

### 2. 以最小上下文收集证据

- 从全局 PRD 和映射到的帮助页开始，帮助页位于 `ref/Docs/CADToolBox/...`。
- 大范围搜索前，先阅读追踪表和一到两个相邻、已完成度较高的场景。
- 用 `rg` 查找当前实现、测试和文档措辞惯例。
- 优先读取足以解释当前行为的最小文件集合。

### 3. 编写或更新场景文档

- 保持三份文档同步：
  - `docs/prd/scenarios`
  - `docs/plan/scenarios`
  - `docs/qa/plans`
- 项目 Markdown 文档统一使用简体中文，沿用仓库现有的中文文件命名和章节命名风格。
- PRD 必须保持诚实：
  - 明确区分帮助页目标态和当前 MVP / 当前仓库边界
  - 明确 `In Scope` 与 `Out of Scope`
  - 写清验收标准和已知差距
- PLAN 必须面向实施：
  - 现状摘要
  - 技术方案
  - 实施阶段
  - 风险
  - 完成定义
  - 验证命令
- QA 计划必须可执行：
  - 关键路径
  - 边界场景
  - 环境
  - 测试数据
  - 自动化验证命令
  - 可接受的已知差异

### 4. 实现最小闭环

- 保持与当前 CadToolOnline 架构一致，不要凭空发明全新框架。
- 尽量复用现有模式：
  - Ribbon 入口
  - 创建面板和属性面板
  - 模型树节点
  - 选择联动
  - 导入导出契约
  - 删除和 `Esc` 行为
- 优先交付最小可用闭环，而不是追求帮助页截图级一致。
- 若 `tdd-workflow` 可用，且当前变更以逻辑契约为主或风险较高，则按其 red-green-refactor 方式实施；否则也必须在结束前补齐回归测试。

### 5. 完成前验证

- 先运行最小但有意义的自动化测试。
- 如果改动了 webview，运行 `pnpm build:webview`。
- 如果改动了 extension 侧，运行 `pnpm build:extension`。
- 仅在改动面足够大时再扩大验证范围。
- 如果某个应跑的验证无法执行，必须在最终回复里明确说明。

### 6. 以场景为中心收口

- 最终总结至少说明：
  - 创建或更新了哪些文档
  - 主要代码与测试改动
  - 实际运行了哪些验证命令
  - 是否存在阻塞项或刻意缩减范围
- 只有在代码和测试确实支持时，才能宣称与帮助页目标态完全对齐。
